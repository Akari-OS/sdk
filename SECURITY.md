# Security Policy — AKARI App SDK

AKARI App SDK はセキュリティを真剣に受け止めている。脆弱性を発見したら、
責任ある方法で報告してほしい。

---

## 脆弱性の報告方法

### ⚠️ 公開 Issue を立てないこと

公開 Issue に脆弱性情報を書くと、修正される前に悪用される可能性がある。
必ず**プライベートな方法**で報告してほしい。

### 報告チャネル

**優先**: 該当リポジトリの Security タブ →「Report a vulnerability」を使う。
GitHub 上で暗号化された報告が送れて、追跡もしやすい。
（例: `https://github.com/Akari-OS/sdk/security/advisories/new`）

**代替**: メールで報告 → security@akari-oss.app

---

## 報告に含めてほしい情報

- 影響を受けるパッケージとバージョン（例: `@akari-os/sdk@0.1.0` または `@akari-os/app-cli@0.2.1`）
- 脆弱性の種類（例: XSS, RCE, 権限昇格、認証バイパス、情報漏洩等）
- 脆弱性が影響する環境（MCP-Declarative Tier / Full Tier / 両方）
- 再現手順（最小限の PoC があると助かる）
- 想定される影響範囲
- 可能であれば修正案

---

## 対応プロセス

1. **受領確認** — 72 時間以内に受領を確認する
2. **初期評価** — 1 週間以内に影響の初期評価を返す
3. **修正開発** — 重大度に応じて修正を開発
4. **リリース** — 修正版をリリース
5. **公開情報開示** — 修正リリース後、CVE / Security Advisory として公開

---

## 重大度の目安

| レベル | 例 | 対応速度 |
|---|---|---|
| **Critical** | RCE、認証バイパス、信頼できない入力からのコード実行 | 即日〜数日 |
| **High** | XSS、コマンドインジェクション、権限昇格 | 1〜2 週間 |
| **Medium** | CSRF、情報漏洩（限定的）、不完全な入力検証 | 2〜4 週間 |
| **Low** | 軽微な設定ミス、ベストプラクティス違反 | 次回リリース時 |

---

## スコープ

本ポリシーは `@akari-os/*` で配布される **SDK パッケージの脆弱性** に適用される。

### 対象

- **SDK 配布パッケージ**:
  - `@akari-os/sdk` — コア型定義 + Phase 1 runtime
  - `@akari-os/app-cli` — 雛形生成・開発・配布ツール
  - `@akari-os/schema-panel` — MCP-Declarative UI レンダラー
  - その他公開パッケージ（`npm` に publish されているもの）

- **依存関係の脆弱性**:
  - `package.json` の direct / indirect 依存で発見された脆弱性
  - npm audit が検出するセキュリティアラート

### スコープ外

- SDK を利用するアプリ（`@akari-os/app-cli` で生成されたアプリ）の脆弱性は、アプリの開発者が責任を持つ
- サードパーティの fork / plugin
- 古いブランチ / 廃止されたバージョン
- 未公開の開発版（main branch の未リリース code）

## 謝辞

責任ある開示をしてくれた研究者には、修正リリース時のリリースノートに
お名前を記載する（希望しない場合は匿名）。

**ありがとう**。あなたの報告が AKARI App SDK を安全にする。
